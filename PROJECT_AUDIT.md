# 本地项目审计与改造路线

## 当前结构

- 核心应用集中在 `index.html`，包含样式、静态数据、认证逻辑、所有页面组件、路由和图表初始化。
- 数据库结构定义在 `setup.sql`，已经覆盖用户、目录数据、课程进度、考试、任务、带教、KPI、评估、通知等核心表。
- 文档比较完整，但代码层目前仍属于“可演示、难维护”的单文件阶段。

## 已接入的真实能力

- Supabase Auth 登录、注册、会话恢复。
- `users` 表资料读取与创建。
- 基于 `userProfile.role` 的导航和页面可见性控制。

## 仍在使用 Mock/静态数据的部分

- 目录数据目前是“Supabase 优先，本地默认数据兜底”，尚未完成真实目录表导入。
- 待审批事项、资源中心内容仍是静态展示。
- 部分模块仍是“Supabase 优先 + 本地兜底”，尚未完成真实生产数据导入。
- 多处按钮只有提示或弹窗，没有持久化保存。

## 主要风险

- `index.html` 单文件过大，改一个功能容易影响其他页面。
- 认证逻辑直接散落在页面里，后续接更多表时会快速失控。
- 页面显示“已连接”只代表配置存在，不代表业务数据真实可用。
- 前端题库和任务长期硬编码，后续很难由业务人员维护。
